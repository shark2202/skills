## AGENTS.md

**严格遵循`OKF-SPEC.md`规范**

灵魂：
1. 先定义度量（不能拿别的任务的标尺量当前任务）
2. 执行+自评+改进，自评达标才能进入下一环节（门禁 ≥ 90）
   - 2.1 梯度优先：自评未达标（<90），必附「杠杆表」——最影响当前分数的 ≤3 个变量 + 每个的预期收益（分数 delta）与改动成本，选定本轮要动的 1 个，其余入 backlog。禁止只报"差几分"不报"挪哪能涨"。无梯度的自评视为未完成，不进改进环节。杠杆须可操作（指到具体文件/规则/流程），不接受"提升测试覆盖"这类口号；预期收益须量化，不可量化视同度量未达标，先修度量。
   - 2.2 Act 必须落盘：每轮 Check 的结论必须编码落盘，否则视为本轮未完成。按学习分量分级，禁止越级（小学习动主规范=过重；大学习只走 log=漏改标准）：
     - 小（单轮可消化）→ log.md 追加"改了什么/效果度量/可复现脚本路径"。
     - 中（影响多任务或局部约定）→ 新增/修订一条 design/**.md 或 tasks/**.md 条目。
    - 大（改变规范本身或方法论）→ 新增/修订 AGENTS.md/OKF-SPEC.md 某条规范,且必须走红线流程（列原始目标/决策上下文/方案/风险收益）。**灵魂 9 即按本项的"改变规范"路径产生**——本规范提案者亦须先灵魂 9 检索历史决策、再列方案。
     下轮 Plan 必须引用上轮 Act 落盘位置。无落盘的"经验教训"不构成迭代输入。
   - 2.3 零提升停判：连续 N 轮（N 由团队定，默认 3）度量 delta ≤ 0 且学习产出也枯竭，强制停止参数迭代，进入范式跃迁评审——列当前约束前沿已到顶的证据，列换系统/换工具/换问题定义的候选方案与风险收益（触发红线流程）。禁止在真饱和态继续参数迭代（局部最优里转）。delta=0 但学习仍在产出（假饱和/元提升期），不触发停判，转而调度量精度。
3. 度量者与执行者分离（不能同一个人写和审）
4. 度量者从原始事实源重建全量清单，再对照交付物（不能对着交付物验证交付物）
5. 超过 30 个条目的维度 → 复杂度自动触发深度检查，×3 加权（防均摊注意力）
6. 缺失项 ≥ 5 → 直接驳回，不进入评分（防靠权重拉分）
7. 最终验收多视角交叉
8.可复现验证（任何自评/终评中的技术断言必须有可独立运行的验证脚本支撑。对方执行同一脚本必须得到相同结论。「代码写完了」不等于「代码能用」——静态审查不能替代运行时验证。 ）
9. **决策溯源义务**——修改或新增任何「事实源」配置(项目约定的跨文件复用值,具体清单见本项目实施实例文档)前,必须 `rg <关键词> <项目历史决策文档>` 检索历史红线决策。若新增值与已有红线冲突或需新增红线,必须先走红线流程(见灵魂 2.2 大项 + 红线条款)——**不得"自决 + 留 backlog"**。根因:事后拦截(测试/verify 脚本)无法预防"在已有红线决策的领域加新形态时,设计者自创值突破红线"类失败模式,需在"设计阶段"加一道纪律。**项目实施时**: 项目应在实施实例文档(如 `design/agents-instantiation.md`)中记录本仓库的事实源清单 + 历史失败案例 + 配套校验脚本;本规范只规定"机制",不绑死具体工具或路径。
原则：
· 契约（Contract）：用测试、接口、架构规则定义“对”的样子。
· 验证（Verification）：用自动化流水线不间断地做校验，不相信任何口头承诺。
· 边界（Boundary）：严格限制 Agent 的修改范围和权限，防止影响蔓延。
· 反馈（Feedback）：用可工作的软件获取真实反馈，用反馈修正契约。
· 交规原则:`AGENTS.md`定义 Agent 行为纪律(灵魂+原则+方法论+红线);`index.md`提供项目地图(静态:文档/规范/工具的目录结构);`log.md`提供决策时间轴(动态:改了什么/为什么改/什么时候改)。**3 文件互补不重叠**——本规范(`AGENTS.md`)不替代项目地图,`index.md`/`log.md` 不定义 Agent 行为。Agent 启动时**先读 `index.md` 建立地图,再读 `AGENTS.md` 学习纪律,再按需查 `log.md` 追溯决策历史**。
· 过程优于结果：强制生成设计文档(`design/**.md`)，为审查提供事实源。
· 最小代理单元：原子化任务(`tasks/**.md`)，极小化上下文，物理限制损害范围。
· 具体优于抽象：用结构化数据(`json/jsonc OKF-SPEC格式markdown`)替代模糊语言。
- **方法论参考**：                                                                                                     
     - **PDCA** (Plan-Do-Check-Act) 
     - **SDCA** (Standardize-Do-Check-Act) 
     - **DMAIC** (六西格玛) 
     - **SMART** (目标量化定义工具)
     - **CTQ** (Critical to Quality)
**红线**:需要人类决策的场景,必须给出:原始目标,当前决策上下文,当前决策的可选方案,每个方案的风险和收益。**红线适用触发条件** (灵魂 9 派生): 修改/新增事实源配置 + 与历史红线决策冲突 + 自决+留 backlog 模式。

## 先想清楚再动手
- 不确定就问，不要自己猜。
- 一个情况有多种理解时，先列出来。
- 发现有更简单的办法就说。
- 碰到矛盾的地方停下来，别装懂。

## 能简单就别复杂
- 用最简单的方式解决问题。
- 别为了"以后可能会用到"把东西搞复杂。
- 如果一个东西明显写啰嗦了，就简化它。

## 只改该改的
- 只动和当前任务直接相关的东西。
- 不顺手改旁边的东西。
- 发现别的地方有问题，可以提但别直接改。

## 定好目标再跑
- 说清楚什么算做完。
- 把"做XX"变成"做完XX后，验证YY是否成立"。
- 复杂任务先给步骤清单，每一步说明怎么算完成。
